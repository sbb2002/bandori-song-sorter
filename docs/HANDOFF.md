#### 스코어링 설계 결정 (확정 — 검토 후 현행 공식 유지)
소표본 보정의 과페널티 우려를 **실데이터로 재검토**한 결과, **현행 공식(0 방향 지수 수축) 유지** 확정. Bayesian 평균 수축·신뢰구간 동점·Empirical Bayes 등 대안은 **미채택**.
- 근거 ①: 소표본(평가곡 n≈τ) 밴드를 **의도적으로 낮게** 주는 게 맞음 — 4곡만 평가한 밴드를 50곡 평가한 밴드와 동일 취급하는 건 부적절.
- 근거 ②: 어느 정도 평가된 밴드끼리(예: n≈22 vs 50)는 둘 다 `w≈1.000` → 같은 비율=같은 점수, 보정 자체가 사실상 무효 → 현행이 이미 의도대로 동작.
- ※ 여기서 `n`은 **사용자가 평가한 곡 수**(출시 곡수 아님). 부분평가로 베테랑 밴드가 과소평가되는 문제는 **공식 수정이 아니라 UI 투명성으로 해결**(아래 #3).
- 데이터 확인: 풀평가 시 13밴드 중 10밴드가 `n≥23 → w≈1.000`. τ 근처는 various_artists(n=5) + 1곡 스텁 2개(ikka·millsage)뿐.

### 2. 자동화 — youtube_rss GitHub Actions 크론 (미구현)
`tools/youtube_rss.py`를 GitHub Actions 크론으로 → 신곡을 **검토용 PR**로 올리기.
- **사용자 방침**: Actions 한도 확인 후 문제없으면 **매일 AM 04:00(KST) 1회**. 크론이 뽑은 후보가 *진짜 신곡/풀버전인지* 판별(false positive)이 과제 — 검증 아이디어 필요.
  - 참고: 이 repo는 **public이라 Actions 분 무료·사실상 무제한**(매일 13밴드 RSS는 ~1분). cron은 UTC 기준이라 04:00 KST = `0 19 * * *`. GitHub 스케줄은 정각 보장 안 됨(고부하 시 수십 분 지연).
- 현재 판별(이미 구현): 영상ID(`known_ids`+`rss_seen.json`) · 정규화 곡명(`norm_name`, 괄호·구두점 제거) · `variant_tag`(TV Size/Short/live/instrumental 제외) 3중 + 곡명당 1개 collapse + inbox 검토 게이트.
- **추천(미구현)**:
  - **제시=PR**: Actions가 신곡을 yaml에 append한 단일 브랜치+PR 자동생성 → 사람이 diff·링크·길이 보고 머지/기각. 검증이 PR 리뷰로 흡수(자동 반영 금지).
  - **상태 영속화**: `rss_seen.json`이 현재 `.gitignore` → Actions는 매 실행 깨끗한 체크아웃이라 seen 없음 → 같은 후보 매일 재생성. seen(+사람이 기각한 `ignore` 목록)을 **git 추적으로 전환**해야 "한 번 기각=재제시 안 함" 성립.
  - **풀버전 신뢰도**: youtube_rss에 영상 길이(watch `lengthSeconds`) 추가 → `variant_tag`가 못 거른 짧은 버전을 길이로 2차 필터, 후보 표에 길이 표기. `variant_tag`에 `movie/anime ver·edit·medley` 패턴 보강.
  - ※ 데이터품질 툴과 oEmbed·길이 스크랩 로직 공유 가능 → 공통 유틸로 묶기.

### 3. ux-02 후속: Download 이미지 밴드별 2채널 막대(선호도+신뢰도) — **구현 완료 · 시각 검토 대기**
부분평가 시 평가곡이 적은(n≈τ) 밴드는 베테랑이어도 점수가 눌리므로(위 #0 결정의 수용된 귀결), 각 밴드에 선호도 막대와 신뢰도(투명도)를 함께 표시해 "점수 낮음 = 덜 평가함"을 인지하게 함.

**구현 내용** (`d58c89a` 후속, `feature/ux-02` 최신 커밋 — ⚠️ **아직 push 안 함, 아래 ‘push 대기’ 참조**):
- **`core.confidence(n, tau)=1-exp(-n/τ)` 헬퍼 추가** — 점수 수축·UI 투명도가 공유하는 단일 정의(n≤0→0). `bandScores`도 이를 사용하도록 정리. 테스트 2건 추가 → **25/25 통과**.
- **`script.js buildCaptureDOM`** 밴드 블록(nameRow 아래, 티어 5막대 위)에 2채널 막대 행 추가:
  - **막대 길이 = 선호도** `max(0, R_k)/4` (R=`bandScores().raw`, 0~1). "그 밴드를 얼마나 높게 매겼나".
  - **막대 투명도 = 신뢰도** `0.15 + 0.85·w(n)` (n=`bandScores().n`, 기존 히트맵 `0.18+0.82` 패턴과 동형, 최저 0.15로 가독 확보). 적게 평가 → 흐림.
  - **막대 색** 골드 `#ffd06b`(= score 숫자색, 시각 연결). 두께 14px, 풀폭. 트랙 `#1e1e2a`.
  - 우측 작은 글씨 `n / 밴드곡수`(밴드곡수=`dedupedByBand[b].length`).
  - 숫자(실제 점수 `Score_k`)는 nameRow에 기존대로 표기(미평가 `—`).
- 점수·1위 선정 **불변**(설명 전용).

**직관 예**: 3곡 전부 최애 → 막대 풀 길이지만 유령처럼 흐림 + 숫자 2.30 = "좋아하지만 아직 불확실". 다 평가한 최애 → 풀 길이+선명. 다 평가한 평범 → 짧고 선명.

**실측(풀평가=곡수 기준 투명도)**: 13밴드 중 11밴드 `alpha≈1.000`(완전 선명), 1곡 스텁 millsage·ikka `alpha=0.361`(흐림), various_artists(5곡) `0.796`. → "다 평가하면 다 선명, 부분평가해야 흐려진다" 의도대로. 부분평가: n=3→0.639, n=5→0.796, n=10→0.951.

**검토 후 조정 가능**(사용자 "일단 봐야 판단" — 브라우저 Download로 확인 후): ① 막대 색(현 골드는 score와 통일되나 "호" 티어색과 동일 → 혼동 시 변경) ② 최저 투명도 0.15(낮춰 "유령" 강조 / 높여 가독) ③ 막대 두께·위치 ④ `n/곡수` 표기 유지 여부.

**폐기된 대안(되돌리지 말 것)**:
- ❌ 막대=`w(n)`: 값 불직관 + 밴드별 도달 상한 다름.
- ❌ 막대=완성도 `n/N`: 1곡 밴드가 100%로 떠 신뢰성 착시.
- ❌ 막대=`Score/(4·w(N))`(선호+표본 합성): 한 막대에 두 개념 → "주먹구구"식 불직관 → 2채널로 분리해 해결.
