## Auto-data-loader에 대한 고찰

* 기능

자동 데이터 등록기는 아래와 같은 파이프라인으로 동작한다.
    1) 신곡 감지    : [서버, 로컬] Youtube RSS를 통해 신곡을 감지.
    2) 신곡 캐시화  : [로컬] yt-dlp를 통해 내려받아 오디오 캐시함.
    3) 신곡 분석    : [서버, 로컬] librosa 등을 통해 음원에 대한 피쳐 추출 및 분석 결과 데이터 보관
    4) 신곡 업데이트 : [서버, 로컬] main에 해당 데이터만 푸쉬하여 업데이트함.


* 문제점

2)번 과정은 yt-dlp를 이용하는데, 데이터베이스 서버로 등록된 IP는 아주 높은 확률로 차단되어있다. 이는 실험적으로 확인되었다.
따라서 물리적으로 상시전원이 들어온 기계를 쓰지 않는 한 이 파이프라인의 완전자동화는 어려울 것으로 사료된다.


* 대안

    - Alt A: 1[서버]-2[로컬]-3[로컬]-4[로컬] : 로컬환경에서 분석까지 가능한 경우 [채택]

    - Alt B: 1[서버]-2[로컬]-3[서버]-4[서버] : 로컬환경에서 분석이 불가능한 경우


* 코멘트

    - 신곡감지 : RSS 피드백으로부터 신곡이 감지되었을 때 현재 데이터에 있는지 검사. 만약 동일 곡 데이터가 있다면 패스, 아니라면 신곡이 떴다고 텔레그램으로 알림을 줄 것.
    
    - 신곡캐시화~신곡업데이트 : 알람을 보고 사용자가 직접 로컬에서 명령어를 실행하여 해당 작업을 수행. 아무래도 음원이 서버로 전송되는 것을 막는 것이 좋아보이기 때문.

    - 이 파이프라인의 full-auto로 만드는 방법은 전혀 없을까? ip 우회없이, 음원 파일의 로컬-서버 간 이동없이 할 수 있는 방법이 필요함. 없다면 현행 방법을 고수해야 함.


## telegram 명령어에 대한 고찰

* 사용 서버 : github actions

* 문제점 

    - 명령어는 모두 잘 작동함. 그러나 actions의 특성 상 한번 트리거되어야 5분 정도 살아있음. 이 기간동안만 명령어를 수행할 수 있음.

    - 알람에 대한 명령어가 필요한가?
        - 솔직히 필요없음. 정 쓴다면 pause, resume, status, help 정도는 괜찮음. 트리거된 타이밍에 가장 처음 실행되는 식이지만, 느리더래도 필요에 따라 동작을 제어할 수 있기 때문.
        - detect 명령어는 트리거된 시점에서 앞서들어온 명령어 처리 후 자동적으로 수행하는 명령임. 따라서 이 명령어는 굳이 없어도 될 것 같음.

* 결정 : detect 명령어의 deprecated 처리 (help에서 관련 내용 제거), 작동방식을 앞서 들어온 명령어 우선 수행 후 detect 동작 수행. 만약 앞서 들어온 아직 수행하지 않은 명령어들 중에 'pause, resume'이 순서대로 1쌍을 이룬다면 서로 상쇄되는 역할이기 때문에 해당 pause와 resume은 무효처리하기.
