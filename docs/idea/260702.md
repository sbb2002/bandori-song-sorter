## 💬 설계 논의 코멘트 (Claude, 2026-07-02)

> 아직 구현/커밋 없음. 방향 검토용 메모. **핵심 결정 확정됨(하단 "결정 확정" 참조) — B안(전곡 확대) + 로컬 벌크/CI 소량 + fail-soft.**

### "API화"의 의미 정정
- HTTP API/백엔드를 만드는 게 아님. **기존 파일은 이동하지 않고**, 새 `actions/` 폴더에 **오케스트레이터(메인 스크립트)** 를 두고 각 기능을 모듈처럼 하나씩 `import`/호출. 그 메인 스크립트를 Actions가 실행.
- 이 레포는 백엔드 서버가 없는 정적 사이트라 상시 HTTP 서버는 부적합. 크론은 "API 호출"이 아니라 러너에서 스크립트를 직접 실행하는 것 → **필요한 건 API 제작이 아니라 워크플로우 배선**. 기존 스크립트는 이미 `--propose`/`main(argv)` 등 CLI 엔트리포인트를 가짐.

### 현재 크론 단계 분해 (`rss.yml` → `youtube_rss.py --propose`)
- **워크플로우 레이어(rss.yml, 6 step):** 트리거(cron 04:00 KST + dispatch) → checkout(fetch-depth 0) → setup-python → `pip install pyyaml` → git identity → `run: youtube_rss.py --propose`.
- **스크립트 레이어(7단계):**
  1. 원장 읽기 `gh_pr_states()` — rss/<id> PR 상태(=이미 다룬 곡)
  2. 기존 데이터 로드 `load_existing()` — songs/*.yaml → known id/name (순수)
  3. 후보 산출 `collect_candidates()` — 밴드별 Topic RSS fetch + 필터 + 길이컷 → candidates/drops/health (네트워크·순수)
  4. 스테이징 `open_pr()` — 곡당 브랜치·yaml 수술삽입·commit·push·`gh pr create` (쓰기)
  5. 이벤트 로깅 `append_events()` — staged/dropped/anomaly → rss_events.jsonl
  6. 헬스 알람 `open_health_issue()` — 파싱 이상 시 gh issue
  7. 로그 커밋 `commit_log()` — rss_events.jsonl만 main에 `[skip ci]` push
- **모듈 경계 인사이트:** **1–3 = 판단/데이터(부수효과 없음)**, **4–7 = 행동/기록(git·gh 쓰기)**. 오케스트레이터의 모듈 경계가 여기 그대로 대응. cluster 모듈의 입력 접점 = **3단계 산출 candidates(band·name·url)**.

### Cluster 단계를 CI에서 돌리는 것 — 판정
- **저작권(산출물): 문제 없음.** 푸쉬되는 건 오디오가 아니라 파생 수치(contrast·mode → x·y 좌표). 러너의 임시 오디오는 폐기. 결과물은 저작물이 아니라 측정값.
- **툴체인 부담: 없음.** ffmpeg는 ubuntu 러너 기본 제공. librosa/numpy는 wheel 설치(콜드 1~2분, pip 캐시 시 수십 초, 잡 1회 비용). 분석 연산은 곡당 ~1초, 증분이면 무시 가능. 공개 레포는 Actions 분당 무제한.
- **진짜 난점 2가지:**
  1. **yt-dlp가 CI 데이터센터 IP에서 봇차단/429** — 크론이 자주 깨질 수 있는 유일 지점.
  2. **증분 좌표 병합 로직 필요** — 현재 `build_perceptual_map.py`는 매니페스트(songs_top10.csv) + 전체 캐시를 **전량 재빌드**. "URL 하나 받아 좌표 산출 후 audio_map.json에 머지"하는 증분 경로는 신규 작성 필요. (BAND_OVERRIDES·y_shift 큐레이션 상수는 코드에 있어 보존)

### 다운로드 신뢰성 — 통상 해법
- **① CI 성공률 올리기(약→강):** 최신 yt-dlp 유지 → 재시도/백오프(`--retries`/`--sleep-requests`, 429용) → 클라이언트 로테이션(`player_client`) → **버너(일회용) 계정 쿠키**(본계정 금지, "봇 확인"에 직접적) → PO token → 레지덴셜 프록시(유료·회색지대, 취미엔 과함).
- **② 다운로드를 CI 밖으로:** 셀프호스티드 러너(레지덴셜 IP) / 로컬에서 추출 후 좌표만 커밋.
- **권고:** RSS가 이미 "seen 원장 없이 매번 재계산(idempotent recompute)" 철학 → cluster도 **fail-soft**로(실패 곡은 스킵·크론 안 죽음·다음 실행 재시도) 짜면 신뢰성이 하드→소프트 요구로 내려감. 1순위=fail-soft, 부족 시 버너 쿠키/클라 로테이션, 그래도 안 되면 셀프호스티드 러너. 프록시/PO token은 취미 규모엔 과함.
- **불변 사실:** 자동 다운로드 자체가 YouTube ToS 위반(로컬·CI 동일) → 계정 리스크는 버너로 격리, 산출물이 파생 수치라는 점이 실무적 방어선.

### 3번(main push)
- 별도 구현 대상 아님. "최종적으로 여기까지 가면 좋겠다"는 목표. 단, 현재 RSS의 human-in-the-loop 게이트(PR 머지=승인, index.html 커밋=별도 게이트)를 자동화가 대체하는 트레이드오프임을 인지.

### 결정 확정 (2026-07-02, 사용자 응답 반영)
- **구멍 1 → B안(전곡 확대)** ✅ : 음원맵 스코프를 top-10×10(100곡) → 전 카탈로그로. 구현 순서·비용은 `docs/working/spec/audio-map-fullscale.md`(①전체660 / ②밴드당 캡N 중 택). 매니페스트를 `songs/*.yaml`에서 생성(dedup=vid) + `build_perceptual_map.py`에 `--manifest` 추가, `audio_full` 캐시로 빌드. (곡 수 부족 밴드 various_artists5·ikka1·millsage1은 중심점 생략/제외)
- **크롭 = 전곡** ✅ : 기존 60초 고정창(45~105s) 폐기 → 전곡 로드(axes.md §2.1 "위치 탐지 없이 통계로 포착"). 채택 축 contrast·mode는 전곡 통계라 60s와 값이 달라짐 → **기존 캐시(60s) 무효, 전곡 재수집 필요.**
- **오디오 캐시 마련 = 로컬 벌크 + CI 소량 + fail-soft** ✅ :
  - 벌크(전곡): 로컬(집 IP)에서 yt-dlp 스로틀 수집 → 분석 후 폐기. CI 봇차단 회피.
  - 신곡(월 <5곡): GitHub Actions에서 감지 곡만 임시 캐시 → 분석 → 폐기.
  - CI 다운로드는 fail-soft(실패 곡 스킵·크론 안 죽음·다음 실행 재시도) — RSS의 idempotent recompute 철학 계승.
- **y_shift/overrides 동결** ✅ (조건부): 특정 밴드(morfonica 등)는 feature로 지각을 못 잡아 큐레이션으로만 보정 → 상시 재튜닝 불필요. **단 97→660 스케일업 시 `zscale` mean/std가 재계산돼 전 좌표 이동 → "전곡 벌크 빌드 결과 육안 승인 직후"에 상수 확정·동결, 그 다음부터 폐기.** (전곡 빌드 = 마지막 튜닝 순간)
- **구멍 3(yt-dlp breakage 알람)** : 별도 알람 시스템으로 대응 예정 → **이번 단계 스킵.**

### ⭐ 증분 경로의 핵심 요구 (폐기 모델을 z-score 좌표와 양립시키기)
좌표는 모집단 mean/std로 z-score된 값 → 오디오를 폐기하고도 신곡을 **전체 재다운로드 없이** 얹으려면:
- **전곡 빌드 시 정규화 파라미터(contrast·mode의 mean·std + x/y_shift + overrides)를 `audio_map.json`에 저장.**
- CI 신곡 경로 = `신곡 오디오 1개 → raw contrast/mode → 저장된 mean/std로 z변환 → 좌표 append`(오디오 즉시 폐기). 월<5곡이면 재z-score 없이 **동결 스케일 append**로 충분(기존 좌표 흔들림 무시 가능).
- 현재 `build_perceptual_map.py`는 매번 전체 캐시에서 mean/std를 재계산할 뿐 저장 안 함 → **이 파라미터 저장이 증분 경로의 선결 조건.**
