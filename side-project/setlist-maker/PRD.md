# setlist-maker — PRD

> `draft.md`(사용자 초안)를 기반으로 작성. 아키텍처 상세는 이 문서의 스코프 밖(별도 설계 문서로 이월).

## 1. 배경 및 문제 정의

`bandori-song-sorter`(EMOI-MAP 등, 정적 사이트)는 오디오 피처 분석 기반의 "구경형" 콘텐츠였다.
umami 통계상 방문자가 사실상 끊긴 상태이며, 그 원인이 마케팅 부족인지 니즈 자체의 부재인지는
확정할 수 없지만, "구경하고 끝나는 콘텐츠"는 구조적으로 재방문 유인이 약하다는 가설을 세웠다.

반면 지금까지 660곡 규모로 축적한 오디오 피처(BPM·조성·에너지·음색)는 "구경용"이 아니라
**실용적으로 반복 소비될 수 있는 형태**(세트리스트 구성)로 재활용할 여지가 있다. 이 PRD는
`bandori-song-sorter`와는 별개의 신규 앱으로, 하모닉 믹싱과 감성 기반 자연어 요청을 결합한
플레이리스트 메이커를 만드는 것을 목표로 한다.

## 2. 목표 및 성공지표

**목표**: 사용자 재방문성이 높은 앱을 만든다(draft.md 원문).

**성공지표(정량 — umami 기준, 파일럿 공개 후 4주 측정)**:

| 지표 | 정의(umami 측정 방법) | 목표치 |
|---|---|---|
| 활성화율 | 방문자 중 플레이리스트를 1개 이상 생성한 비율(`playlist_created` 이벤트 발생 방문자 / 전체 방문자) | ≥ 40% |
| 7일 재방문율 | 첫 방문 후 7일 이내 재방문한 방문자 비율(umami returning visitors) | ≥ 15% |
| 세션당 생성 수 | `playlist_created` 이벤트 수 / 세션 수 | ≥ 1.2 |
| 연속 재생률 | 생성된 플레이리스트 중 3곡 이상 연속 재생된 비율(`song_advance` 이벤트 2회 이상) | ≥ 40% |
| 절반 완주율 | 목표 재생시간의 50% 이상 재생된 플레이리스트 비율(`playlist_half_played` 이벤트) | ≥ 25% |

- 목표치는 초기 가설치이며, 파일럿 실측 데이터 확보 후 보정한다.
- 측정을 위해 umami 커스텀 이벤트 3종(`playlist_created`·`song_advance`·`playlist_half_played`)
  계측이 파일럿 구현 요구사항에 포함된다(6번 외부 연동 참조).
- **베이스라인**: bandori-song-sorter의 동일 기간 재방문율(현재 사실상 0)을 비교 기준으로 삼아,
  "구경형→실용형 전환이 재방문을 만든다"는 1번의 가설을 위 지표로 검증한다.

파일럿 통과 기준은 draft.md의 "[참고]" 절대로 핵심 기능이 정상 동작하는지이며, 위 지표는 파일럿
이후 정식 운영(및 부수 기능 투자) 판단에 사용한다.

## 3. 타깃 사용자 / 핵심 유스케이스

밴드리 음악을 즐겨 듣는 사용자가 특정 상황·기분에 맞는 긴 재생 흐름을 직접 곡을 고르지 않고
자연어 한 문장으로 얻고 싶어하는 경우.

- 유스케이스 A: "오늘 퇴근하면 주말 시작이야~ 기분이 절로 좋아지는 플레이리스트를 만들어줘"
  → 밝은 무드 곡 위주, 에너지가 점차 고조되는 약 1시간 세트리스트.
- 유스케이스 B(파생): "차분하게 집중할 수 있는 곡으로, 30분만" → 에너지 낮게 유지, 짧은 러닝타임.

draft.md의 핵심아이템 3종 대응: "플레이리스트 메이커"는 4번(파일럿 범위), "플레이리스트 공유"는
5번(다음 단계)에서 다룬다. "**사용자 친화적 앱**"은 특정 기능이 아니라 전 단계 공통 설계 원칙으로
둔다 — 자연어 한 문장만 입력하면 결과가 나오는 최소 입력 동선을 유지하고, 설정(5번-1)은 전부
합리적 기본값으로 동작해 설정을 몰라도 쓸 수 있어야 한다.

## 4. 파일럿 범위 (Must-have)

1. 자연어 요청 입력창 제공.
2. LLM(OpenRouter) 기반 자연어 의도 분석 → 무드/에너지 방향(예: 밝음/어두움, 시작 에너지 레벨) 추출.
3. 1차 후보곡 필터링 — 파일럿에서는 밴드 전체 대상(밴드 체크박스 필터는 다음 단계로 이월).
4. 에너지 진행 로직: 사용자 지정 단계 수(N, 기본값은 오픈 퀘스천)에 따라 단계별 에너지 레벨을
   지정하고, 각 단계 내에서 (a) 해당 단계 에너지 레벨에 맞는 곡 선택 + (b) 직전 곡과 하모닉하게
   이어지는 조성(Camelot Wheel 인접 규칙)의 곡을 우선 선택.
5. 총 재생시간 목표(기본 60분, 사용자 조정 가능)에 맞춰 곡 수를 자동 산정.
6. 생성된 세트리스트를 유튜브 iframe으로 순서대로 자동 재생(다음 곡 자동 전환).

**파일럿 범위 제외**: 유튜브 재생목록으로 계정 저장·타인 공유(OAuth 연동)는 다음 단계로 이월한다.
파일럿은 "생성 + 순차 재생"까지만 검증한다.

## 5. 다음 단계 (Should/Could-have)

draft.md의 "부수 기능" 1)~3)을 전부 반영한다.

1. **설정 UI** (UI 형태 미정):
   - 에너지 레벨 조절 — 플레이리스트를 시간순 N단계로 나누고, 단계별 지정 시간 또는 곡 수, 해당
     단계 에너지 레벨을 사용자가 직접 지정.
   - 밴드 체크박스 필터 — 1개 이상 밴드 선택 시 해당 밴드 내에서만 검색. 전체 선택 또는 빈 선택은
     ALL로 처리. 에너지 구간별로도 개별 설정 가능.
2. **요청 큐 제어** — 동시 사용자가 몰릴 경우 OpenRouter rate limit에 걸릴 위험이 있어, 요청
   큐잉/순서 제어가 필요. 파일럿 단계는 동시 사용자가 적을 것으로 예상되므로, 큐 없이 요청당
   단순 실패 처리(재시도 없음)로 시작하고, 트래픽이 실제로 늘어나는 시점에 큐를 도입하는 순서를
   제안한다(제약사항 7번과 교차 참조).
3. **백엔드 호스팅 정책** — 큐 제어와 OpenRouter API 키 보관을 위해 백엔드 서버가 필요하며,
   가급적 무료 티어를 사용하고 유휴 시 sleep(cold start 허용)되는 방식을 선호한다(기술 고려사항
   8번과 교차 참조).
4. 유튜브 재생목록 계정 저장·공유(OAuth 연동) — 4번에서 파일럿 범위 밖으로 명시된 항목.

## 6. 데이터 요구사항

기존 `bandori-song-sorter` 레포에서 재사용 가능한 자산이 이미 충분히 확보되어 있다.

| 데이터 | 위치 | 비고 |
|---|---|---|
| 유튜브 URL | `src/content/cluster/songs_full.csv` (`idx,band,song,url`, 660행) | `url`이 전부 `https://youtu.be/<video_id>` 포맷 — video_id는 파싱으로 추출(신규: 추출 헬퍼 필요) |
| 무드 매칭 피처 | `side-project/genre-features/song_features_with_proxies.csv` | `mode_score`(장/단조)·`energy_proxy`·`acousticness_proxy`·`instrumentalness_proxy`·`tempo_excerpt`(BPM), 660행 null 0건 |
| 조성(하모닉 믹싱) | 위 CSV의 `key` 컬럼 | `Amaj`/`Amin` 등 24종 — Camelot Wheel 매핑 테이블 신규 필요 |
| 보조 피처 | `src/content/cluster/audio_map.json`(`songs[i] = {band, song, url, x, y, bpm, energy, shape}`) | 이미 계산된 `bpm`·`energy`·음색 시그니처 `shape` — 무드 필터링에 참고 가능 |

**신규로 필요한 것**:
- video_id 추출 헬퍼(URL 파싱).
- key(`Amaj`/`Amin` 표기) → Camelot Wheel 코드 매핑 테이블.
- 표본 부족 밴드(ikka_dumb_rock·millsage 등 n=1) 처리 정책 — 파일럿에서는 이런 밴드를 후보군에서
  제외하거나, 최소 표본 기준(예: n≥10) 미달 밴드는 자동 제외하는 규칙 필요(오픈 퀘스천).

draft.md가 필요 데이터로 언급한 "오디오 캐시(분위기 매칭 분석용)"에 대해: 피처 자체는 이미 전곡
추출·커밋되어 있으나(위 표), **현재 피처는 EMOI-MAP 펄스 애니메이션 용도로 설계·추출된 것**이라
setlist-maker의 분위기 매칭 목적에 그대로 충분한지는 추가 조사가 필요하다. 조사 결과 무드 매칭에
새 피처(또는 다른 파라미터의 재추출)가 필요하다고 판단되면 원본 오디오 캐시 접근이 다시 필요해진다
— draft.md가 오디오 캐시를 필요 데이터로 유지한 이유. 파일럿은 우선 기존 피처로 시작하되, 피처
적합성 검토를 파일럿 중 확인 항목으로 둔다(9번 오픈 퀘스천). 전곡(660) 오디오 캐시는 별도 로컬에
있고 이 로컬에는 일부만 있다는 운영 제약도 재추출 시 고려해야 한다.

**외부 연동(API) 요구사항** (draft.md "필요 API" 대응):
- **OpenRouter API** — 자연어 의도 분석(4번-2). 키는 백엔드에서 보관(8번).
- **YouTube iframe Player API** — 순차 자동 재생(4번-6). 파일럿에서는 재생목록 저장을 이월했으므로
  Data API·OAuth가 불필요하고, 다음 단계(5번-4)에서 Data API 연동이 필요해진다.
- **umami** — 2번 성공지표 측정. 신규 앱에 스크립트 설치 + 커스텀 이벤트 3종 계측 필요.

## 7. 제약사항 및 리스크

- **LLM 해석의 모호성**: 같은 자연어 문장도 사용자마다 다른 무드로 해석될 여지가 있다 — 파일럿
  단계에서는 결과 세트리스트에 "왜 이렇게 골랐는지" 간단한 설명을 붙여 기대와 어긋날 때 사용자가
  바로 확인할 수 있게 하는 것을 권장(구현 여부는 오픈 퀘스천).
- **key 컬럼의 신뢰도 미검증**: 현재 `key` 값은 음정 추정 알고리즘의 출력이며, 정확도가 별도
  검증된 바 없다 — 하모닉 믹싱 추천이 이따금 부자연스러운 조합을 낼 수 있음을 감안해야 한다.
- **밴드별 표본 편중**: poppin_party 115곡 ~ ikka_dumb_rock/millsage 1곡까지 편차가 크다 — 특정
  무드 요청에 특정 밴드 곡만 반복 추천될 위험이 있다(6번 데이터 요구사항 참조).
- **동시성/레이트리밋**: 5번의 "요청 큐 제어"가 다음 단계로 이월된 상태에서는, 동시 사용자가
  몰리면 OpenRouter 호출이 실패할 수 있다 — 파일럿 단계는 실패 시 단순 에러 메시지로 처리.
- **저작권**: 오디오 파일 자체를 저장·배포하지 않고 유튜브 iframe 재생만 사용하므로,
  `bandori-song-sorter`와 동일한 안전 범위 내에 있다.

## 8. 기술 고려사항 (방향성만 — 상세 설계는 별도 문서)

LLM(OpenRouter) API 키 보관과 향후 요청 큐 제어를 위해서는 정적 사이트가 아닌 백엔드 서버가
필요하다. 호스팅은 무료 티어를 우선하고, 유휴 시 sleep(cold start 허용)되는 방식을 선호한다
(draft.md의 명시적 제약). 구체적인 백엔드 스택·플랫폼 선정은 이 PRD의 스코프 밖이며, 별도
아키텍처 문서에서 다룬다.

**클린 아키텍처(파일럿 이후 구현 시 필수)**: 도메인 로직(무드 해석 결과 스키마, 에너지 진행·
하모닉 믹싱 선곡 규칙)은 외부 서비스에 직접 의존하지 않도록 하고, LLM 호출은 도메인이 정의한
포트(인터페이스) 뒤에 어댑터로 격리한다. **OpenRouter 내 모델 교체는 물론 벤더 자체 교체**(예:
Anthropic/OpenAI 직접 호출로 전환)**도 어댑터 1개 교체만으로 가능한 구조여야 한다.** 선곡 규칙은
LLM 출력(구조화된 무드/에너지 파라미터)만 입력받는 순수 함수로 유지해, LLM 없이도 단위 테스트가
가능하게 한다. (같은 워크스페이스 전례: investbot의 adapter(프로토콜 계층) → api(비즈니스 로직)
분리 패턴.)

## 9. 오픈 퀘스천

- 에너지 단계(N) 기본값과 최대/최소 범위.
- LLM 벤더(OpenRouter 내 모델) 및 예상 비용.
- 무료 백엔드 호스팅 플랫폼 선정(예: Render/Railway/Fly.io 등 — 미결정).
- 요청 큐 제어 구현 방식과 도입 시점(트래픽 임계치).
- 파일럿 단계에서 동시 사용자 규모 가정치.
- 표본 부족 밴드(n<10 등) 제외 기준의 정확한 임계값.
- LLM 해석 결과에 대한 사용자 대상 설명(왜 이 곡을 골랐는지) 노출 여부.
- 기존 피처(EMOI-MAP 펄스 용도로 추출)의 무드 매칭 적합성 — 부족하면 어떤 피처를 오디오 캐시에서
  재추출할지(6번 참조).
